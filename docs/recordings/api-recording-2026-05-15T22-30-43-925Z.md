## Fachliche Einordnung

Diese Aufnahme erschliesst den Bestellbereich der Warenwirtschaft. Sichtbar ist ein Workflow vom Vorgang/Artikelkontext zur Bestellvorschlag-Erzeugung, danach Suche der Bestellvorschlaege, Umwandlung in eine Bestellung und Oeffnen der Bestelldetails inklusive Lieferant, Positionen, Wareneingangssuche und Eingangsrechnungssuche.

Fuer die Companion-App ist wichtig: Bestellung beginnt nicht isoliert bei `orders`, sondern kann aus einem Vorgang beziehungsweise einer Artikelposition entstehen. Der beobachtete Workflow fuehrt ueber `order-proposals` und erst danach zu `orders/from-proposal`.

## Rekonstruierter Bestellworkflow

1. Vorgangs-/Artikelkontext herstellen
   - Kundensuche, neuer Vorgang, Position/Preislogik und mehrere `PUT /apigateway/sales/salesprocesses/{uuid}`.
   - Relevante bekannte Endpunkte: `POST /apigateway/pricingservice/sales-positions`, `POST /apigateway/sales/salesprocesses/calculate-prices`, `POST /apigateway/wawiservice/stock-items/count-article-quantities`.

2. Artikel fuer Bestellung vorbereiten
   - `GET /apigateway/article-tenant/articles/{uuid}`
   - `GET /apigateway/article-tenant/articles/{uuid}/price-data`
   - `GET /apigateway/article-tenant/articles/{uuid}/supplier-assignments`
   - `GET /apigateway/article-tenant/articles/{uuid}/computed-order-value/{id}` mit `supplierId` und `unit`
   - `POST /apigateway/supplier/suppliers/list`
   - `GET /apigateway/supplier/suppliers/search`

3. Bestellvorschlag erzeugen
   - `POST /apigateway/wawi/order-proposals` -> `201`
   - Request enthaelt unter anderem `articleId`, `supplierId`, `filialeId`, `orderQuantity`, `orderQuantityUnit`, `orderValue`, `articleOrigin` und Bezug zum Vorgang.

4. Bestellvorschlaege suchen und summieren
   - `GET /apigateway/wawi/order-states`
   - `GET /apigateway/wawi/producers`
   - `GET /apigateway/wawi/storage-locations`
   - `GET /apigateway/supplier/suppliers`
   - `POST /apigateway/wawi/order-proposals/search/sums`
   - `POST /apigateway/wawi/order-proposals/search`

5. Vorschlag in Bestellung umwandeln
   - `POST /apigateway/wawi/order-proposals/to-order`
     - Request: `{ includeAll: boolean; selections: string[]; filters: null }`
   - `POST /apigateway/wawi/orders/from-proposal`
     - Request: `{ proposals: { includeAll: boolean; selections: string[]; filters: null } }`
   - Danach wird die erzeugte Bestellung geladen.

6. Bestellung oeffnen und Kontext hydrieren
   - `GET /apigateway/wawi/orders/{uuid}`
   - `GET /apigateway/wawi/orders/{uuid}/positions`
   - `GET /apigateway/supplier/suppliers/{uuid}`
   - `GET /apigateway/supplier/suppliers/{uuid}/addresses`
   - `GET /apigateway/supplier/suppliers/{uuid}/contacts`
   - `GET /apigateway/supplier/suppliers/{uuid}/contact-opportunities`
   - `GET /apigateway/supplier/suppliers/{uuid}/customers`
   - `POST /apigateway/wawi/order-arrival/search`
   - `POST /apigateway/wawi/incoming-invoices/search`

## Auffaelligkeiten

- `GET /apigateway/article-tenant/articles/{uuid}/details/{uuid}` lieferte einmal `404`. Der Workflow lief trotzdem weiter. Wahrscheinlich ist das ein optionaler Detailkontext oder ein Request mit nicht vorhandener Detail-ID; vor produktiver Nutzung sollte dieser Pfad gezielt erneut aufgezeichnet werden.
- `orders/from-proposal` antwortete mit `200`, nicht `201`. Die erzeugte Bestellung wird anschliessend ueber `GET /apigateway/wawi/orders/{uuid}` geladen.
- Die Bestellansicht hydriert viele Lieferanten-Unterdaten separat. Fuer einen Companion sollte das als ein zusammengefasster `getOrderContext(orderId)`-Workflow gekapselt werden.
