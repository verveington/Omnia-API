## Fachliche Einordnung

Diese Aufnahme ist der bisher belastbarste Realtest fuer den Zielprozess. Sie zeigt nicht nur Lesen und Export, sondern einen produktiven Ablauf aus Omnia heraus:

1. Bestellvorschlag anlegen.
2. Bestellvorschlag in eine Bestellung umwandeln.
3. Bestellung verarbeiten und dabei ein Bestelldokument erzeugen.
4. PDF-/Maildaten fuer den lokalen Mailclient vorbereiten.
5. Bestellung im Wareneingang suchen.
6. Wareneingang mit Liefernummer, Lieferdatum, Menge, Filiale und Lagerort buchen.

Der Test bestaetigt die bisherige Annahme: Die Companion-App kann diesen Ablauf technisch als eigene Fassade abbilden, muss aber die gefaehrlichen Omnia-Details kapseln. Besonders `orders/from-proposal` und `order-arrival/book` duerfen nicht als freies API-Formular exponiert werden.

## Rekonstruierter Ablauf

### 1. Bestellvorschlag erzeugen

- `POST /apigateway/wawi/order-proposals`
- Status: `201`
- Request enthaelt `articleId`, `supplierId`, `filialeId`, `quantity`, `unit`, `orderValue`, `salesProcessId`, `customerId` und `kind`.
- Beobachtete `kind`: `OWN_ORDER_PROPOSAL`.

Implikation: Wenn die Companion-App spaeter Bestellvorschlaege erzeugen soll, braucht sie einen validierten Artikel-, Lieferanten-, Filial- und Vorgangskontext. Fuer V1 bleibt das noch ausserhalb des Read-only-Exports.

### 2. Vorschlaege fuer Bestellung vorbereiten

- `POST /apigateway/wawi/order-proposals/to-order`
- Status: `200`
- Request-Muster:

```json
{
  "includeAll": false,
  "selections": ["<proposalId>"],
  "filters": null
}
```

Implikation: Die Companion-Fassade sollte immer mit expliziter Auswahl arbeiten. `includeAll` ist fuer produktive Automatisierung riskanter, weil die Filterlage schwerer sichtbar ist.

### 3. Bestellung aus Vorschlag erzeugen

- `POST /apigateway/wawi/orders/from-proposal`
- Status: `200` in erfolgreichen Faellen, `500` in zwei Fehlerfaellen.

Erfolgreiches Request-Muster:

```json
{
  "proposals": {
    "includeAll": false,
    "selections": ["<proposalId>", "<proposalId>"],
    "filters": null
  },
  "supplierId": "<supplierId>"
}
```

Fehlerhaft beobachtetes Request-Muster:

```json
{
  "proposals": {
    "includeAll": false,
    "selections": ["<proposalId>"],
    "filters": null
  },
  "resolveUnitMismatch": true
}
```

Die Fehlerantwort enthielt nur `500 Internal Server Error` auf `/orders/from-proposal`, keine fachliche Fehlermeldung.

Implikation: Fuer die Companion-App ist Lieferantengruppierung nicht nur fuer Exporte wichtig, sondern auch fuer produktives Bestellen. Die Fassade sollte pro Lieferant arbeiten und `supplierId` explizit setzen. Unit-Mismatch darf nicht blind geloest werden; wenn Omnia dafuer einen instabilen Pfad nutzt, muss die Companion-App den Fall als klaerungspflichtig markieren.

### 4. Bestellung laden und Positionen pruefen

- `GET /apigateway/wawi/orders/{uuid}`
- `GET /apigateway/wawi/orders/{uuid}/positions`
- Status: `200`

Implikation: Nach `orders/from-proposal` sollte die Companion-App die erzeugte Bestellung erneut laden und gegenpruefen, ob Positionen, Menge, Lieferant und Wert zur gewaehlten Vorschlagsgruppe passen.

### 5. Bestellung verarbeiten

- `POST /apigateway/wawi/orders/{uuid}/process-order`
- Status: `200`
- Request-Muster:

```json
{
  "orderState": null,
  "orderDate": "2026-05-16T15:18:42.712Z",
  "markProposalsAsOrdered": true,
  "printOptions": {
    "stationery": true,
    "prices": true,
    "emailHeader": false,
    "emailFooter": false,
    "showSalesProcessNumber": true
  }
}
```

- Response enthaelt `orderDocumentId`.

Implikation: Die produktive Aktion sollte in der Companion-App als klarer Schritt modelliert werden: Bestellung erzeugen, dann Bestellung verarbeiten. `markProposalsAsOrdered` ist fachlich relevant und sollte nicht implizit versteckt werden.

### 6. PDF-/Maildaten vorbereiten

- `POST /apigateway/wawi/orders/{uuid}/email`
- Status: `200`
- Request-Muster:

```json
{
  "createMailFile": true,
  "documentIds": ["<orderDocumentId>"]
}
```

- Response enthaelt `subject`, `receivers`, `documents` und `mailFileId`.

Implikation: Omnia bereitet PDF-/Maildaten vor. Der eigentliche Versand erfolgt ueber den lokalen Mailclient, nicht serverseitig ueber Omnia. Die Companion-App darf diesen Schritt nicht als "E-Mail versendet" darstellen.

### 7. Wareneingang suchen

- `POST /apigateway/wawi/order-arrival/search`
- Status: `200`

Implikation: Der Wareneingang sollte aus einer gefilterten, vom BFF erzeugten Suche kommen. Die UI braucht Auswahl nach Bestellung, Lieferant, Filiale und Status, aber keine rohe Filter-JSON.

### 8. Wareneingang buchen

- `POST /apigateway/wawiservice/order-arrival/book`
- Status: `200`
- Request-Muster:

```json
{
  "orderNumber": "<orderNumber>",
  "deliveryDate": "2026-05-16T17:19:34.838Z",
  "deliveryNr": "<deliveryNumber>",
  "filialeId": "<filialeId>",
  "storageLocationId": "<storageLocationId>",
  "applyToAllFiliale": false,
  "quantity": 1,
  "salesProcessStatusId": null,
  "versorgungsStatusId": null,
  "comment": null,
  "editorId": "<editorId>",
  "zeroRemainingAmount": false,
  "performGoodsReceipt": true,
  "filteredSelection": {
    "includeAll": false,
    "selections": ["<arrivalPositionId>"],
    "filters": null
  }
}
```

- Response zeigt die Bestellung anschliessend mit `orderArrivalBookingState: "DONE"`.

Implikation: In diesem Realtest hat ein einzelner `book`-Call mit `performGoodsReceipt: true` den Wareneingang direkt abgeschlossen. In einer frueheren Aufnahme gab es zusaetzlich `book-recorded`. Die Companion-App sollte beide Varianten kapseln und anhand der Omnia-Antwort entscheiden, ob ein zweiter Schritt noetig ist.

## Fehler und Randfaelle

- `orders/from-proposal` lieferte zweimal `500`, wenn der Request den `resolveUnitMismatch`-Pfad statt einer expliziten `supplierId`-Gruppierung nutzte.
- `GET /apigateway/article-tenant/articles/{uuid}/details/{uuid}` lieferte einmal `404`. Das war im Test nicht fatal; Artikelgrunddaten konnten ueber andere Artikelendpunkte geladen werden.
- `DELETE /apigateway/wawi/order-proposals/{uuid}` kam im Test vor und war erfolgreich. Die Companion-App sollte Loeschen nicht automatisch ausfuehren, sondern nur nach expliziter Benutzeraktion.

## API-Design-Implikationen

Fuer die voll integrierte Companion-App sollte das BFF folgende fachliche Aktionen anbieten:

- `createOrderProposal(input)`
- `previewSupplierOrder(proposalIds, supplierId)`
- `createOrderFromSupplierProposals(supplierId, proposalIds)`
- `processOrder(orderId, printOptions)`
- `prepareOrderMail(orderId, documentIds)`
- `searchOrderArrivals(filters)`
- `bookGoodsReceipt(orderNumber, deliveryData, selection)`
- `completeGoodsReceiptIfRequired(orderIdOrOrderNumber)`

Guardrails:

- Immer pro Lieferant bestellen.
- Vor Bestellung PZN-, Lieferanten-, Mengen- und Einheitenstatus pruefen.
- `supplierId` bei `orders/from-proposal` explizit setzen.
- Unit-Mismatch als Benutzerklaerung behandeln, nicht automatisch ueber `resolveUnitMismatch`.
- Nach jedem schreibenden Omnia-Call die Bestellung beziehungsweise den Wareneingang erneut lesen.
- PDF-/Mailvorbereitung nicht als serverseitigen Versand bezeichnen.
- Wareneingang nur mit expliziter Positionauswahl buchen.

## Naechste Umsetzungsschritte

1. BFF-Live-Adapter fuer authentifizierte Omnia-Requests stabilisieren.
2. Read-only Bestellplattform um einen geschuetzten Aktionsmodus erweitern.
3. Lieferantengruppen als harte Bestellgrenze verwenden.
4. `createOrderFromSupplierProposals` mit Preflight-Checks implementieren.
5. Wareneingangsmodul auf `order-arrival/search` und `order-arrival/book` mappen.
6. Audit-Log je Benutzer speichern: gewaehlte Vorschlaege, erzeugte Bestellung, Dokument-ID, Maildatei-ID und Wareneingangsbuchung.
